## Task Description

Write unit tests for `postgresAdapter.js` that verify the full adapter interface (run/get/all/exec/transaction/prepare/checkpoint/close) matches the contract of the existing SQLite adapters. These tests serve as the contract guarantee that all repositories can rely on the adapter.

## Context

The SQLite adapters have informal test coverage via repository tests. The PostgreSQL adapter needs explicit interface tests because its async/sync difference, placeholder translation, and pg-specific return shapes can introduce subtle bugs. These tests should run against a real PostgreSQL (the docker-compose.test.yml from Domain 8).

## Implementation Steps

### Step 1: Create test file
```bash
mkdir -p tests/unit
# tests/unit/db-adapter-postgres.test.js
```

### Step 2: Test run() method
- INSERT: verify `{ changes: Number, lastInsertRowid: Number }` return shape
- UPDATE/DELETE: verify `changes` is correct count
- Verify `changes` and `lastInsertRowid` are Numbers, not strings (pg returns strings by default)

### Step 3: Test get() and all()
- `get` with no rows returns `undefined`
- `get` with one row returns the row object
- `all` with no rows returns `[]`
- `all` with N rows returns array of N row objects
- BOOLEAN columns return actual booleans, not 0/1
- JSONB columns return JS objects, not strings

### Step 4: Test exec()
- `exec("CREATE TABLE t (id INT)")` succeeds
- Multi-statement exec works (e.g., `CREATE TABLE a; CREATE TABLE b`)

### Step 5: Test transaction(fn)
- Successful fn: changes are persisted
- Thrown error: changes are rolled back, error re-thrown
- Return value of `fn` is returned by `transaction`

### Step 6: Test prepare(sql)
- `prepare` returns `{ run, get, all }`
- Each method has the same normalized return shape as the non-prepared versions
- Calling multiple times reuses the prepared statement

### Step 7: Test checkpoint() and close()
- `checkpoint()` does not throw (no-op for PG)
- `close()` ends the pool; subsequent queries throw an error

### Step 8: Test placeholder translation
- SQL with 0, 1, 3, 5 `?` placeholders with matching param count
- Wrong param count should throw a clear error

### Step 9: Add graceful skip
```js
const POSTGRES_URL = process.env.POSTGRES_URL;
if (!POSTGRES_URL) test.skip("POSTGRES_URL not set", () => {});
```
