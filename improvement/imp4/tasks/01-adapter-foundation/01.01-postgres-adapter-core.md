## Task Description

Implement `src/lib/db/adapters/postgresAdapter.js` — the PostgreSQL implementation of the database adapter interface. This is the foundational piece that all other domains depend on. The adapter must expose the same `run`, `get`, `all`, `exec`, `transaction`, `prepare`, `checkpoint`, `close`, and `raw` API as the SQLite adapters, but use `pg` (node-postgres) with connection pooling under the hood.

## Context

The existing adapter system has 4 SQLite adapters (`betterSqliteAdapter`, `bunSqliteAdapter`, `nodeSqliteAdapter`, `sqljsAdapter`). All expose the same surface and are wired up via `src/lib/db/driver.js`. The PostgreSQL adapter must slot into this same selection chain, becoming the first choice when `POSTGRES_URL` is set. Key differences vs. SQLite:
- File-based -> connection-based (use `pg.Pool`)
- Sync operations -> async (pg's `client.query` is promise-based)
- No WAL -- `checkpoint()` becomes a no-op
- `?` placeholders must be translated to `$1, $2, ...`
- `changes`/`lastInsertRowid` come back as strings from pg -- must be coerced to Number
- JSONB columns return JS objects directly (no parse needed)

## Implementation Steps

### Step 1: Add pg to dependencies
```bash
# In project root
npm install pg
# Verify
grep '"pg"' package.json
```

### Step 2: Create the adapter file with skeleton
```js
// src/lib/db/adapters/postgresAdapter.js
import pg from "pg";

export async function createPostgresAdapter(config = {}) {
  const { connectionString, poolSize = 20, ssl = false } = config;
  const pool = new pg.Pool({ connectionString, max: poolSize, ssl });
  // ... helpers, return adapter object
}
```

### Step 3: Implement run(sql, params)
- Translate `?` placeholders to `$1, $2, ...` via `_convertParams(sql, params)`
- Call `pool.query(sql, params)`
- Return `{ changes: Number(result.rowCount), lastInsertRowid: ... }`
- Coerce all numeric return values to `Number`

### Step 4: Implement get(sql, params) and all(sql, params)
- `get`: `(await pool.query(sql, params)).rows[0]`
- `all`: `(await pool.query(sql, params)).rows`

### Step 5: Implement exec(sql) for DDL
- Used by `init()` to run schema statements
- `await pool.query(sql)` works for multi-statement DDL (semicolon-delimited)

### Step 6: Implement transaction(fn)
- Acquire client: `const client = await pool.connect()`
- `await client.query("BEGIN")`
- `try { const r = await fn(); await client.query("COMMIT"); return r; }`
- `catch { await client.query("ROLLBACK"); throw; }`
- `finally { client.release() }`

### Step 7: Implement prepare(sql)
- Cache a `PreparedStatement` per SQL string in a `Map`
- Returns `{ run, get, all }` with normalized return shapes
- Use stable names for pg prepared statements

### Step 8: Implement checkpoint() and close()
- `checkpoint()`: `return` (no WAL in PG)
- `close()`: `await pool.end()` and clear statement cache
- Register cleanup on `SIGINT`/`SIGTERM`/`beforeExit`

### Step 9: Set adapter metadata
- `adapter.driver = "postgres"`
- `adapter.kind = "postgres"`

## Success Criteria

- [ ] `createPostgresAdapter({ connectionString, poolSize, ssl })` returns a connected adapter
- [ ] `adapter.driver === "postgres"`
- [ ] `run` returns `{ changes: Number, lastInsertRowid: Number }` for INSERT
- [ ] `get` returns `undefined` when no rows, otherwise the row object
- [ ] `all` returns `[]` when no rows, otherwise an array of row objects
- [ ] `exec` runs multi-statement DDL without error
- [ ] `transaction(fn)` commits on success, rolls back on throw
- [ ] `prepare(sql)` returns a reusable statement object
- [ ] `checkpoint()` does not throw (no-op)
- [ ] `close()` ends the pool; subsequent queries throw
- [ ] `?` placeholders in SQL are converted to `$1, $2, ...` correctly for 0-10 params

## Notes

- Use `pg` (node-postgres) -- the de-facto standard
- The `_convertParams` helper replaces all `?` (outside of string literals) with `$N`
- Pool size default 20 is reasonable for multi-replica with PgBouncer; for direct connections 10-20 is fine
- JSONB serialization: pg-node automatically stringifies objects passed as parameters, and parses JSONB columns on retrieval
