## Task Description

Replace the read-modify-write pattern for the `totalRequestsLifetime` counter with a single atomic SQL statement. The current pattern is racy under concurrent flushes.

## Context

The counter is currently updated with:
1. `SELECT value FROM _meta WHERE key = 'totalRequestsLifetime'`
2. Compute `newValue = parseInt(value) + N`
3. `UPDATE _meta SET value = ? WHERE key = 'totalRequestsLifetime'`

Under concurrent flushes, two flushes can read the same baseline, both compute the same new value, and one's update is lost. The fix is a single atomic SQL statement.

## Implementation Steps

### Step 1: Read current counter logic
```bash
grep -n "totalRequestsLifetime\|_meta" src/lib/db/repos/usageRepo.js
```

### Step 2: Replace with atomic SQL
```sql
UPDATE _meta 
SET value = (value::bigint + $1)::text 
WHERE key = 'totalRequestsLifetime' 
RETURNING value;
```

### Step 3: Implement in usageRepo
```js
const result = await db.get(
  "UPDATE _meta SET value = (value::bigint + $1)::text WHERE key = 'totalRequestsLifetime' RETURNING value",
  [entries.length]
);
const newTotal = parseInt(result.value, 10);
```

### Step 4: Verify with 100 parallel calls
- All 100 calls complete
- `totalRequestsLifetime` === 100 (the invariant from `tests/unit/db-concurrent.test.js`)

## Notes

- The `value::bigint + $1::bigint` cast is required because _meta.value is TEXT
- The final `::text` cast puts it back to TEXT for storage
- RETURNING is a PostgreSQL extension to standard SQL
- For SQLite, the existing read-modify-write pattern is fine (SQLite serializes writes)
- The metaStore helper should be updated to expose `incrementCounter(key, by)` for reuse
