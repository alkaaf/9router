## Task Description

Create `deploy/k8s/postgres-statefulset.yaml` and `deploy/k8s/postgres-pvc.yaml` for self-hosted PostgreSQL deployment. Uses the official `postgres:16-alpine` image, with resource requests/limits, liveness/readiness probes, and a 50Gi PVC.

## Context

The 9Router can run with a self-hosted PostgreSQL (StatefulSet) for development/small deployments, or use a managed service (RDS, Cloud SQL, Aiven) for production. This task creates the self-hosted manifests. Managed PG deployment is documented but not provided as manifests (operators typically use Terraform/Pulumi for that).

## Implementation Steps

### Step 1: Create postgres-pvc.yaml
```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
spec:
  accessModes: ["ReadWriteOnce"]
  resources:
    requests:
      storage: 50Gi
  # storageClassName: standard  # uncomment for cloud providers
```

### Step 2: Create postgres-statefulset.yaml
```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres
  replicas: 1
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16-alpine
          ports:
            - containerPort: 5432
          env:
            - name: POSTGRES_DB
              valueFrom:
                secretKeyRef: { name: postgres-secret, key: POSTGRES_DB }
            - name: POSTGRES_USER
              valueFrom:
                secretKeyRef: { name: postgres-secret, key: POSTGRES_USER }
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef: { name: postgres-secret, key: POSTGRES_PASSWORD }
            - name: PGDATA
              value: /var/lib/postgresql/data/pgdata
          resources:
            requests: { cpu: 100m, memory: 256Mi }
            limits:   { cpu: 500m, memory: 1Gi }
          livenessProbe:
            exec: { command: ["pg_isready", "-U", "$(POSTGRES_USER)"] }
            initialDelaySeconds: 30
            periodSeconds: 10
          readinessProbe:
            exec: { command: ["pg_isready", "-U", "$(POSTGRES_USER)"] }
            initialDelaySeconds: 5
            periodSeconds: 5
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:
    - metadata: { name: data }
      spec:
        accessModes: ["ReadWriteOnce"]
        resources: { requests: { storage: 50Gi } }
```

### Step 3: Apply to a test cluster
```bash
# If using kind/minikube:
kubectl apply -f deploy/k8s/postgres-pvc.yaml
kubectl apply -f deploy/k8s/postgres-statefulset.yaml
# Wait for pod ready
kubectl wait --for=condition=ready pod -l app=postgres --timeout=120s
# Verify
kubectl exec -it postgres-0 -- psql -U <user> -d 9router -c "SELECT 1"
```

## Success Criteria

- [ ] `kubectl apply -f postgres-pvc.yaml` creates PVC
- [ ] `kubectl apply -f postgres-statefulset.yaml` creates a running PG pod
- [ ] `kubectl exec postgres-0 -- psql ...` connects successfully
- [ ] PVC binds to a volume, data persists across pod restarts
- [ ] Liveness and readiness probes configured

## Testing Checklist

- [ ] Apply manifests to kind/minikube
- [ ] Verify pod reaches Ready state
- [ ] psql connection works
- [ ] Stop and start the pod, verify data is preserved

## Notes

- `subPath: pgdata` with `PGDATA=/var/lib/postgresql/data/pgdata` avoids the "mount over an existing data dir" issue
- For production: use managed PG (RDS/Cloud SQL) instead of StatefulSet
- The Secret referenced (`postgres-secret`) is created in 06.04
- Consider adding `podAntiAffinity` to prevent multiple replicas on the same node
