## Task Description

Add a PostgreSQL test job to `.github/workflows/ci.yml`. SQLite tests run on every commit (the fast default path). PostgreSQL tests run on PRs that touch DB code (adapter, schema, repos, drivers). Use GitHub Actions' `services: postgres` for the PG instance.

## Context

CI should be fast for normal PRs but catch PG regressions on DB-touching PRs. The split: SQLite tests are always run; PG tests are conditionally run based on which files changed. This keeps CI fast and avoids running PG on docs-only PRs.

## Implementation Steps

### Step 1: Read current CI config
```bash
cat .github/workflows/ci.yml
# Identify the existing test job structure
```

### Step 2: Add PostgreSQL service to a new job
```yaml
jobs:
  test-postgres:
    runs-on: ubuntu-latest
    if: |
      contains(github.event.pull_request.changed_files.*.path, 'src/lib/db/') ||
      contains(github.event.pull_request.changed_files.*.path, 'src/lib/db/adapters/') ||
      contains(github.event.pull_request.changed_files.*.path, 'src/lib/db/repos/') ||
      contains(github.event.pull_request.changed_files.*.path, 'src/lib/db/schema') ||
      contains(github.event.pull_request.changed_files.*.path, 'src/lib/db/driver.js')
    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_DB: 9router_test
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
        ports:
          - 5432:5432
        options: >-
          --health-cmd "pg_isready -U test -d 9router_test"
          --health-interval 2s
          --health-timeout 5s
          --health-retries 30
    env:
      POSTGRES_URL: postgres://test:test@localhost:5432/9router_test
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '22' }
      - run: npm ci
      - run: npm run build 2>/dev/null || true
      - run: npm test -- --grep postgres
```

### Step 3: Add a path filter for the PG job
- The `if:` condition uses `contains(...)` for each path
- Or use `dorny/paths-filter` action for cleaner logic

### Step 4: Verify in a test PR
- Create a test PR that touches `src/lib/db/adapters/postgresAdapter.js`
- Verify the PG test job runs
- Create a test PR that only touches `docs/`
- Verify the PG test job is skipped

## Notes

- Use `services: postgres` instead of docker compose for simpler CI
- The `if:` condition should match the migration's actual file paths
- For a simpler implementation, use `dorny/paths-filter` action
- Consider adding a `test:postgres` matrix entry for multiple PG versions (16, 17)
- The PG test job should NOT block merges -- it's a safety net
- Keep SQLite tests as the default fast path; PG is for thoroughness
